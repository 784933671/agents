# 证据采集

调试的本质是"从现象到根因"，而证据是连接两者的桥梁。本技能聚焦如何系统化采集高质量证据，让问题可分析、可交接、可复现。

## 何时使用本技能

- 偶现问题需要捕获下次发生时的现场
- 把 bug 交接给他人（同事、上游、社区）
- 准备最小复现例用于报告或测试
- 建立问题的"证据链"以支撑根因判断

## 证据的四个层次

```
1. 现象描述  → "它偶尔白屏"          （最弱，主观）
2. 截图/录屏  → 可见发生时的画面       （直观但缺内部状态）
3. 日志/快照  → 发生时的运行时数据     （强，可归因）
4. 最小复现  → 可稳定触发的最短路径     （最强，可验证修复）
```

目标是把证据从第 1 层推到第 4 层。

## 采集方法

### 1. 结构化日志

发生问题时的关键上下文：

```js
// 临时埋点：捕获下次复现的现场
window.addEventListener('error', (e) => {
  console.error('[capture]', {
    msg: e.message,
    stack: e.error?.stack,
    url: location.href,
    time: new Date().toISOString(),
    state: window.__APP_STATE__,  // 关键应用状态
  });
});
```

要点：
- 带**时间戳**和**唯一标识**（如 requestId）
- 记录**状态快照**，不只是错误信息
- 区分**发生了什么**和**推断为什么**

### 2. DOM 与状态快照

```js
// 捕获出错时的 DOM 结构
const snapshot = document.querySelector('#app').outerHTML;

// 捕获状态机当前状态
const stateSnapshot = JSON.stringify(store.getState());
```

### 3. 网络请求录制

复现时同步导出 HAR：

- Network 面板 → 右键 → Save all as HAR
- HAR 含完整请求/响应/时序，便于离线分析
- 注意脱敏（token、敏感数据）

### 4. 录屏与操作回放

- 浏览器内置录制（DevTools → Recorder）
- 或用工具录屏 + 操作日志
- 配合时间戳，与日志对齐

## 构造最小复现例

把复杂场景压缩到最简：

### 剥离步骤

```
完整流程: 登录 → 浏览 → 搜索 → 加购 → 结算 → 复现
逐步剔除: 结算 → 加购 → 搜索 → 复现
最小: 登录后直接访问 /checkout → 复现 ✓
```

### 固定变量

- 数据：用确定的数据而非随机
- 环境：固定浏览器版本、网络条件
- 时序：去掉动画、延迟等时序依赖

### 自包含

最小复现应能独立运行：

```html
<!-- 单文件复现例 -->
<!DOCTYPE html>
<html>
<body>
  <div id="app"></div>
  <script>
    // 最小代码触发问题
  </script>
</body>
</html>
```

## 证据交接清单

向他人报告 bug 时，证据应包含：

```
1. 现象: 可观察的客观描述（不带推断）
2. 环境: 浏览器/OS/版本、入口、账号（脱敏）
3. 复现步骤: 编号步骤，每步可独立验证
4. 预期 vs 实际: 明确对比
5. 证据: 截图/录屏/日志/HAR/最小复现
6. 频率: 必现/偶现（注明触发条件）
7. 影响范围: 哪些用户/场景受影响
```

## 反模式

- ❌ 只描述现象不给证据（"它崩了"）
- ❌ 证据带推断而非事实（"因为缓存所以错"——这是结论不是证据）
- ❌ 复现步骤有隐藏前提（"先登录"未说明）
- ❌ 最小复现仍依赖整个项目（非自包含）
- ❌ 日志没保留现场状态（只留了错误信息）

## 原则

1. **事实与推断分离**：证据记录事实，根因另列
2. **可复现**：证据要让他人能验证
3. **自包含**：最小复现不依赖外部未声明条件
4. **脱敏**：对外共享前清理敏感信息
5. **时效**：趁现场还在赶紧采集，事后补不完整
