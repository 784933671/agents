# 热点分析

从性能数据（火焰图、trace、profiler）中系统化定位最值得优化的热点。核心原则：**优化占比最大的瓶颈，而非最容易改的地方**。

## 何时使用本技能

- 有一份性能录制或 trace，需要找出瓶颈
- 面对"整体慢"但不知道具体慢在哪
- 评估多个候选优化点，决定先做哪个
- 向他人解释"为什么是优化这里而不是那里"

## 分析流程

```
1. 选取数据 → 2. 找宽块 → 3. 归因 → 4. 排序 → 5. 给建议
```

### 1. 选取数据

确认分析对象与指标：

- **加载慢**：用 Performance 录制一次加载，关注主线程和网络
- **交互慢**：录制触发交互的操作，关注 INP 相关的长任务
- **请求慢**：用 Network 的 Timing 或服务端 trace
- **运行慢**：用 CPU profiler 或火焰图

数据要代表**真实问题场景**，而非合成场景。

### 2. 找宽块（火焰图解读）

火焰图中，**横向越宽 = 占用时间越长**。按宽度找热点：

```
阅读顺序：
  - 顶层（调用栈顶）：实际执行的函数
  - 底层（调用栈底）：入口
  - 宽块：耗时大户，优先关注

优先级：
  1. 最宽的叶子块 → 自身耗时高
  2. 被反复调用的窄块（累积宽） → 总耗时高
  3. 单次很宽但只调一次 → 可能是必要开销
```

DevTools 的 **Bottom-Up** 视图按"自身耗时"和"总耗时"排序，比火焰图更直接地列出热点。

### 3. 归因

对每个热点回答：

| 问题 | 判断 |
|------|------|
| 自身耗时还是调用栈耗时？ | 看 Self Time vs Total Time |
| 是业务逻辑还是框架开销？ | 看函数归属 |
| 是否可避免？ | 同等场景下基线对比 |
| 调用次数异常？ | 高频小函数累积 |

典型归因：

```
- 长 Layout 块 → 布局抖动（JS 读写交替）
- 长 Parse HTML → DOM 过大或内联脚本
- 长 Evaluate Script → 单个重型计算或大依赖
- 长 Recalculate Style → 复杂选择器或大量元素
- 网络 TTFB 长 → 服务端处理慢
```

### 4. 排序优化点

对每个候选热点评估，按 **收益/成本比** 排序：

| 热点 | 自身耗时 | 优化难度 | 预期收益 | 优先级 |
|------|---------|---------|---------|--------|
| 大接口可缓存 | 600ms | 低 | 高 | P0 |
| 长任务可拆分 | 150ms | 中 | 中 | P1 |
| 资源压缩 | 30ms | 低 | 低 | P2 |

排序原则：
- 先做收益高且难度低的
- 收益低但极简的可顺手做
- 收益高但难度大的单列，评估是否值得

### 5. 给建议

每个建议包含：

```
热点: <函数/阶段>
证据: <占比、调用次数、火焰图位置>
根因: <为什么慢>
建议: <优化方向，不展开实现>
预期收益: <量化>
验证: <如何复测确认>
```

## 常见热点模式

### 主线程长任务

特征：火焰图上一整块红色超过 50ms。

| 模式 | 根因 | 方向 |
|------|------|------|
| 大同步循环 | 算法复杂度或数据量 | 拆分、用 web worker、分页 |
| 重复渲染 | 状态变化触发整树重渲染 | memo、key、状态下沉 |
| 布局抖动 | JS 交替读写 DOM | 批量读写、用 transform |

### 网络热点

特征：Network 中某请求横跨很长时间。

| 模式 | 根因 | 方向 |
|------|------|------|
| TTFB 长 | 服务端处理慢 | 后端优化、缓存 |
| Content Download 长 | 响应体大 | 压缩、分页、字段裁剪 |
| 队列等待 | 并发连接限制 | 减少并发、HTTP/2 |
| 串行请求 | 本可并行 | Promise.all |

### 渲染热点

特征：大量 Layout / Paint / Composite 块。

| 模式 | 根因 | 方向 |
|------|------|------|
| 频繁 Layout | 改了几何属性 | 改 transform/opacity |
| 大面积 Paint | 重绘区域大 | 减少层、缩小重绘区 |
| 层爆炸 | 过多合成层 | 合理使用 will-change |

## 反模式

- ❌ 优化"看着不顺眼"的代码而非热点
- ❌ 只优化自身耗时，忽略累积耗时
- ❌ 没有基线，无法量化收益
- ❌ 一次性改多处，无法归因收益
- ❌ 追求局部极致而忽略整体瓶颈

## 原则

1. **数据驱动**：热点由数据指出，而非直觉
2. **归因到根**：慢的是函数，根因是设计或用法
3. **收益排序**：先做高收益，而非高难度或高趣味
4. **可验证**：每条优化前后都有测量
5. **一次一项**：改一处测一次，避免收益混淆
