# 代码库映射策略

在陌生代码库中建立结构化心智模型的系统方法。核心是"有策略地探索"，而非漫无目的地翻文件。

## 何时使用本技能

- 接手一个陌生项目，需要快速上手
- 对大型代码库做技术评估或尽职调查
- 为团队编写代码库导览或 onboarding 文档
- 定位某个功能的实现位置

## 映射策略

### 策略一：入口追踪法

从启动点倒推整个系统结构：

```
找入口 → 跟初始化 → 看路由/分发 → 摸到各模块
```

入口在哪：

| 类型 | 典型入口 |
|------|---------|
| Web 服务 | main / app.listen / bootstrap |
| CLI | bin/ 下的可执行、yargs/commander 定义 |
| 库 | package.json 的 main/exports |
| 框架应用 | 框架约定的入口（Next.js 的 app/、Django 的 urls.py） |

从入口沿调用链向下，比从目录结构向上猜更高效。

### 策略二：特征定位法

带着具体问题定位：

```
"用户登录怎么实现的？"
  → 搜 login / signin / authenticate
  → 找到 handler
  → 看它调的 service
  → 看它用的 model
```

工具组合：
- **grep / ripgrep**：按关键词搜，快速缩小范围
- **符号跳转**：IDE 的"转到定义/查找引用"
- **调用层级**：谁调了它、它调了谁

### 策略三：差异化阅读

不读全部代码，而是对比阅读：

- **新旧对比**：git log 看近期变更，理解活跃区域
- **正常 vs 异常**：对比 happy path 和 error handling
- **配置 vs 默认**：看哪些行为是可配的，哪些是写死的
- **测试 vs 实现**：测试揭示意图，实现揭示细节

### 策略四：分层概览

按职责分层建立骨架，而非按目录：

```
1. 接口层  对外暴露什么（HTTP / RPC / CLI / 导出 API）
2. 业务层  核心逻辑在哪
3. 数据层  状态如何持久化
4. 横切层  鉴权、日志、错误、配置怎么贯穿
```

每层只需一句话概括职责，细节按需深入。

### 策略五：跟随数据

追踪核心数据结构流转，常比追踪控制流更清晰：

```
User 对象在哪创建 → 在哪校验 → 在哪存储 → 在哪返回
```

数据是名词，函数是动词，沿名词找往往结构更清晰。

## 高效探索工具

| 目的 | 工具 |
|------|------|
| 搜关键词 | ripgrep（快）、grep |
| 看结构 | tree / ls -R、IDE 文件树 |
| 理解依赖 | dep-tree、madge、依赖图插件 |
| 看历史 | git log、git blame |
| 看规模 | cloc、wc -l（找异常大的文件） |
| 看测试覆盖 | coverage 报告（揭示哪些是核心） |

## 建立心智模型

读完探索后，提炼成可记忆的模型：

```
项目 = [核心概念] 在 [边界] 内通过 [流程] 实现 [目标]

例：
  电商系统 = [订单/商品/用户] 在 [多租户隔离] 内
            通过 [下单→支付→履约] 实现 [在线交易]
```

模型要：
- 用项目自己的术语（而非泛化术语）
- 包含边界与约束（不只是"做什么"，还有"不做什么"）
- 标出未理解的部分（坦诚比臆测有价值）

## 反模式

- ❌ 从第一个文件逐个读（线性阅读大忌）
- ❌ 只看实现不看测试（丢失意图）
- ❌ 陷入细节忘了建立骨架
- ❌ 把自己的项目范式强加给目标项目
- ❌ 把"没看懂"包装成"它设计得不好"

## 原则

1. **入口优先**：从启动点或入口倒推
2. **问题驱动**：带具体问题探索，而非通读
3. **先骨架后细节**：先整体后局部
4. **尊重既有约定**：用项目自己的语言
5. **标注未知**：坦诚标记没理解的部分
