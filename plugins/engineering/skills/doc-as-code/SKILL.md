# 文档即代码

把文档当作代码一样管理：同仓、同版本、同 review、可构建可验证。目标是让文档**容易维护**且**不容易过期**。

## 何时使用本技能

- 为项目建立技术文档体系
- 编写或重构 API 文档、架构文档
- 让文档与代码保持同步，减少过期
- 评估现有文档的组织方式是否合理

## 核心原则

```
1. 就近   文档离它描述的东西越近越好
2. 同步   代码变了，文档要能跟着变
3. 可构建 Markdown 是源，渲染是产物
4. 可验证 链接、示例代码、术语能被检查
```

## 文档分层

按读者和用途分层，而非按"什么都写一份"：

| 层 | 读者 | 内容 | 位置 |
|----|------|------|------|
| 项目说明 | 所有人 | 是什么、怎么跑 | README.md |
| 贡献指南 | 贡献者 | 怎么参与、规范 | CONTRIBUTING.md |
| 架构决策 | 维护者 | 为什么这么设计 | ADR / docs/architecture |
| API 参考 | 使用者 | 接口契约 | 从代码生成 / OpenAPI |
| 运行手册 | 运维 | 怎么部署、排障 | docs/runbook |

不同层有不同生命周期，分开维护比塞在一个大文档里更耐用。

## 就近原则

文档离描述对象越近，过期概率越低：

```
❌ 集中式（容易过期）
docs/
└── modules.md           ← 描述所有模块，改代码时容易忘更新

✅ 就近式（跟着代码变）
modules/
├── auth/
│   └── README.md        ← auth 模块的文档就在 auth 旁边
├── order/
│   └── README.md
```

模块/包内的 README 描述本模块，顶层 README 只做索引。

## API 文档：从代码生成

手写 API 文档必然过期，从代码生成才长久：

```
代码注释/类型定义（源）
        ↓ 工具
   API 参考（产物）
```

- TypeScript：TypeDoc / TSDoc
- Go：godoc
- Python：Sphinx autodoc
- REST：OpenAPI / Swagger

注释写在代码里，改代码时顺手改注释。

## 架构决策记录（ADR）

每个重要决策记一份短文档，而非维护一份不断改的"架构总览"：

```markdown
# ADR-007: 选择 PostgreSQL 而非 MongoDB

日期: 2026-07-07
状态: 已采纳

## 背景
需要事务、关系完整性、成熟生态。

## 决策
采用 PostgreSQL 作为主数据库。

## 结果
放弃了 schema-less 的灵活性，获得强一致性。
```

ADR 只增不改（推翻时新增一条标记 superseded）。这比一份反复修改的"架构文档"更能保留"为什么"。

## 可运行文档

让文档里的代码示例能跑，过期就会在 CI 里暴露：

- **代码块测试**：用 lint 工具检查 markdown 里的代码
- **可执行 notebook**：复杂示例用 notebook，CI 跑一遍
- **链接检查**：CI 检查 markdown 里的链接是否有效
- **术语一致性**：关键术语用统一名称

```yaml
# .github/workflows/docs.yml 示例
- name: 检查文档链接
  run: lychee docs/ README.md
- name: 检查代码示例
  run: markdownlint-cli2 docs/**/*.md
```

## README 的最小完备

一份好的项目 README 至少回答：

```
1. 是什么     一句话定位
2. 为什么      解决什么问题，与替代品的差异
3. 怎么跑      安装 + 启动 + 最小使用
4. 怎么参与    贡献入口、规范链接
5. 在哪深入    指向更详细的文档
```

不写"全"，写"够上手"。

## 文档腐烂的征兆与对策

| 征兆 | 原因 | 对策 |
|------|------|------|
| 文档说 A，代码做 B | 文档没跟着改 | 就近放置、从代码生成 |
| 链接 404 | 目标移动了 | CI 链接检查 |
| "TODO: 待补充" 堆积 | 没人补 | 要么补要么删，不留僵尸 |
| 一份文档几千行 | 单文档承载过多 | 按主题拆分 |
| 截图过期 | UI 早变了 | 少用截图，多用文字描述 |

## 反模式

- ❌ 把所有文档塞进一个巨型 README
- ❌ 手写 API 文档（必然过期）
- ❌ 截图替代文字（截图最先过期）
- ❌ "TODO: 后续补充"永远不补
- ❌ 文档和代码分仓（同步成本高）

## 原则

1. **就近**：文档离描述对象越近越好
2. **生成优先**：能从代码生成的就别手写
3. **分层**：不同读者不同文档，而非一份通吃
4. **可验证**：链接、示例、术语能在 CI 里检查
5. **只增不改**：决策用 ADR 记录，而非反复改总览
