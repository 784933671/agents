# 重构护栏

让重构"安全"的一组强制约束。重构的本质是"改结构不改行为"，而护栏确保"行为不变"这条线不被越过。

## 何时使用本技能

- 计划对一段代码做结构改善
- 评估一个重构方案是否安全
- 重构过程中保持节奏，避免失控
- 重构后验证行为确实未变

## 第一护栏：安全网先行

没有安全网的重构是赌博。开始前必须建立行为基线。

### 已有测试

```bash
# 跑全量，确认绿色基线
npm test 2>&1 | tee /tmp/before.log
```

记下基线：多少通过、多少失败（理想是全绿；若已有失败，记下失败清单作为"既有状态"）。

### 无测试 → 特征化测试

不写"应该怎样"的测试，写"现在就是这样"的测试，锁住当前行为：

```js
// ❌ 主观测试（可能和行为不符）
test('add 应该返回数值和', () => {
  expect(add(2, 3)).toBe(5);
});

// ✅ 特征化测试（锁定现状，哪怕现状是 bug）
test('add 当前行为', () => {
  expect(add(2, 3)).toBe(5);
  expect(add(2, '3')).toBe('23');  // 锁定隐式转换，即使不合理
});
```

覆盖关键路径的输入输出即可，不必追求完备。

### 手动基线

连测试都无法写时，至少记录可观察的行为：

```
- 访问 /api/users 返回 [{id:1,name:"a"}]
- 输入 X 触发错误提示 Y
- 操作序列 A→B→C 后页面显示 D
```

这些是重构后必须复现的"行为契约"。

## 第二护栏：原子步骤

每步只做一件事，且每步独立可验证。

### 步骤拆解示例

```
目标：把 200 行的 processOrder 拆分

❌ 一次性重构（危险）
  提取 5 个函数 + 重命名 + 移动文件，一个大 commit

✅ 原子步骤（安全）
  commit 1: 提取 validateOrder 函数        → 测试
  commit 2: 提取 calculateTotal 函数       → 测试
  commit 3: 提取 persistOrder 函数         → 测试
  commit 4: 重命名 processOrder 为 handleOrder  → 测试
  commit 5: 移动到 services/order.ts       → 测试
```

### 步骤独立性检验

每步提交前问：
- 这一步单独能跑吗？
- 这一步的意图能一句话说清吗？
- 出问题能单独回退这一步吗？

任一答"否"，说明步骤太大。

## 第三护栏：一次一事

一个 commit 只做一种变更：

| ✅ 一个 commit | ❌ 混在一起 |
|---------------|-----------|
| 只重命名 | 重命名 + 修 bug |
| 只提取函数 | 提取函数 + 改逻辑 |
| 只移动文件 | 移动文件 + 改导入风格 |

混在一起会导致：
- 测试失败时分不清是哪部分引起的
- 回退时连带丢弃无关变更
- review 时难以判断每步的安全性

发现"顺手能修"的问题时，**记下来，重构完单独处理**。

## 第四护栏：即时验证

每步执行后立即验证，不等"全做完一起测"：

```
改一步 → 跑测试 → 绿 → 提交
改一步 → 跑测试 → 红 → 回退到上一步，重新拆解
```

红了绝不"再改改试试"。回到绿色提交点，重新设计更小的步骤。

## 第五护栏：行为保持红线

重构绝不能（任何一条触线都不是重构，是改行为）：

- 改变公开 API 签名
- 改变可观察输出（返回值、副作用、错误类型/顺序）
- 偷偷修 bug
- 改变性能特征（除非目标就是性能）
- 改变可观察的日志/事件

发现需要改行为时：**停下重构，开新任务**。

## 提交与回退策略

### 提交粒度

```
每个原子步骤 = 一个 commit
commit message：描述"做了什么重构"，而非"为什么"
```

例子：
```
refactor: extract validateOrder from processOrder
refactor: rename processOrder to handleOrder
refactor: move order logic to services/
```

### 回退机制

```
安全网在 → 红了 → git reset --hard HEAD~1（回退一步）
安全网弱 → 红了 → 仔细评估，可能需要手动恢复
```

保持线性、小步的历史，回退成本才低。

## 反模式

- ❌ "重构顺便把这个 bug 修了"
- ❌ "改完了再写测试"
- ❌ "这一步比较大，但应该没问题"
- ❌ "红了，我再调调，可能就好了"
- ❌ 重构 PR 里夹带新功能

## 原则

1. **安全网先行**：无基线不开始
2. **小步原子**：每步可独立验证
3. **一次一事**：不夹带行为变更
4. **即时验证**：每步立即测
5. **可回退**：任何时刻能回到绿色
