# UX 测试设计

以"用户流程"为主线、以"状态完整性"为兜底的功能测试设计方法。目标是发现真实使用中会暴露的问题，而非穷举边界。

## 何时使用本技能

- 为一个新功能或流程设计测试用例
- 规划 UI 回归测试的范围
- 结构化记录测试中发现的缺陷
- 制定可复用的测试脚本/检查清单

## 测试设计流程

```
1. 梳理用户流程 → 2. 分解步骤与状态 → 3. 设计用例 → 4. 执行与记录
```

### 1. 梳理用户流程

从"用户想完成什么"出发，而非"页面有哪些元素"：

```
流程：用户下单
  登录 → 浏览商品 → 加入购物车 → 填写地址 → 选择支付 → 提交 → 确认
```

每个流程要覆盖：
- 主路径（happy path）
- 异常路径（失败、中断、返回）
- 替代路径（游客下单、换支付方式）

### 2. 分解步骤与状态

每个步骤展开为可测的状态组合：

| 步骤 | 关键状态 |
|------|---------|
| 加入购物车 | 库存不足、已是最大数量、需选规格 |
| 填写地址 | 必填校验、格式校验、异地运费 |
| 提交 | 网络超时、库存刚变、重复提交 |

### 3. 设计用例

用例模板：

```
用例ID: UC-001
流程: 用户下单
前提: 已登录、购物车有 1 件商品
步骤:
  1. 进入结算页
  2. 地址留空
  3. 点击提交
预期: 提交被阻止、地址字段就近报错、错误可读
优先级: 高
```

用例覆盖维度：

- **主路径**：正常流程一遍跑通
- **状态覆盖**：加载/空/错误/成功
- **边界**：极长输入、极多数据、并发操作
- **环境**：不同视口、弱网、断网恢复

### 4. 执行与记录

逐用例执行，发现偏差即记录缺陷。

## 缺陷报告结构

```
缺陷ID: BUG-001
标题: 地址为空时仍可提交订单
用例: UC-001
环境: Chrome 120 / 桌面端
严重程度: 阻断
复现步骤:
  1. 购物车有 1 件商品
  2. 进入结算页
  3. 清空地址
  4. 点击提交
实际结果: 订单被创建，地址为空
期望结果: 提交被阻止，提示填写地址
附件: 截图/录屏
```

严重程度定义：

| 级别 | 含义 | 例子 |
|------|------|------|
| 阻断 | 核心流程无法完成 | 下单永远失败 |
| 严重 | 主要功能受损 | 支付成功但无反馈 |
| 一般 | 次要功能或体验问题 | 错误提示位置不对 |
| 轻微 | 美观/文案问题 | 错别字 |

## 测试设计原则

1. **流程优先**：测用户做的事，而非页面的元素
2. **状态完备**：每个交互都要覆盖四种状态
3. **可复现**：步骤要精确到任何人能复现
4. **不穷举**：聚焦真实风险，不为覆盖率堆用例
5. **回归友好**：用例可脚本化，便于自动化

## 测试脚本组织建议

```
tests/
├── flows/              # 按用户流程组织
│   ├── checkout.spec.ts
│   └── signup.spec.ts
├── components/         # 按组件
│   └── cart.spec.ts
└── fixtures/           # 测试数据与状态
```

按流程组织比按页面组织更接近真实使用，也更容易在重构后保持稳定。
